Q1: FDB decouples the TMS from the SS to separate concerns of concurrency control from durable storage. The TMS handles transaction ordering, conflict detection, and version assignment. The SS handles writing data to disk and serving reads. This separation lets each layer scale and fail independently to give them more resilience and performance. 

Q2: The Resolver component within the TMS is responsible for detecting read-write conflicts between concurrent transactions. The check occurs during the commit process after a client has sent its read/write sets to a Proxy that forwards the transaction's read set to the Resolver, which checks whether any of those keys were written by a transaction that committed after the current transaction's read version. The transaction is only logged if no conflict is found.

Q3: The read version represents a consistent snapshot of the database at a specific point in time, assigned by the Sequencer at the start of a transaction. It makes sure that all reads in a transaction see a consistent view of the data, so all committed writes before that version and none after. Transactions read from a stable state, and FDB guarantees that the final commit will be consistent with that observed state or will be aborted if a conflict is detected. 

Q4: Snapshot reads bypass conflict detection entirely. Performing reads this way does not add keys to the transaction's read conflict set, meaning another transaction writing to those keys will not cause a conflict. Standard reads track every key read and will conflict if a concurrent writer commits a change to those keys before the current transaction commits. We would choose snapshot reads when we need to read data for informational purposes like a report and can tolerate reading potentially stale data, or when we want to avoid spurious conflicts on frequently-read hot keys that the transaction does not depend on. 

Q5: FDB's recovery is cheap because StorageServers continuously pull and apply mutations from LogServers during normal operation, so the durable state on StorageServers is always close to fully up-to-date. When a failure happens, the system only needs to recover the small window of recent mutations from the LogServers rather than replaying a full redo log from the beginning. Because LogServers durably persist every committed mutation in order, and StorageServers already have nearly current state, recovery simply involves replaying a short tail of recent log entries to bring StorageServers fully in sync.
